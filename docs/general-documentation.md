## Networking

### Packets
For both my deserialization and serialization I use std::vectors since I use the flexibility of the vector in both methods various implementations.  Deserialization uses it to shave off the size of the packet and packet ID varints off of the packet before passing the data of the packet to the deserialize method.  The resizing of this vector may be slower with the logical approach of abstraction and removing responsilibity to repare the beginning of the packet within the deserialize method.  It may be faster to keep it one static std::array and re-parse as needed, or even slice array as needed.  For serialization I use the variable size of the vector because of the varying size of strings/JSONs being parsed.  Some of this is mitigated possibly by including some reserve method calls as needed.  Overall the amount of memory reallocations could be drastic here and should be considered in the future if these method calls are consuming a large amount of time.

### Packet Serialization (Outgoing packets)
These outgoing packets must be given their needed data at the time of their construction so that the serialize method does not need to fetch the data during serialization.  These methods are already a mess to look at.

For much of my packet serialization I append a +1 to the packet length.  This is because nearly all of the packet ID's are less than 128 (2^7) which is one varInt.  Regardless we can know the length of the varint as we write these serializations we can append the correct length saving a lot of unnecessary work.

### Packet Deserialization (Incoming packets)
As mentioned above, the use of ```std::arrays``` should be investigated here.

The packet deserialization methods are  closely coupled with the Connection class so that the second the server receives a connection change state packet it is done immediately so as to not create any race conditions between future expected packets and whatever would be handling the task of changing the state at a future time.  I have yet to think of a cleaner way of implementing this at the moment.

I might create a general incoming packet constructor to store a reference to the packets Connection parent so that it does not need to be passed as a argument to the deserialize method.  But I may just be nit picky.

### Packet Ordering
Here I will document my ordering for the packets from when the client first initiates the handshake up until the play state.


```
Handshake:

Client -> Server : Handshake_p [0x00]
- Create the client connection
- transition it to either Status or Login state

Status:

Client -> Server : Status_Request_p [0x00]
- Queue the Status Response packet

Server -> Client : Status_Response_p [0x00]
- Send the JSON with the server data

Client -> Server : Ping_Request_p [0x01]
- Receive timestamp as a long

Server -> Client : Ping_Response_p [0x01]
- Send back the timestamp long

Login:

Client -> Server : Login_Start_p [0x00]
- Receive player login info

Server -> Client : Encryption_Request_p [0x01]
- Send player Encryption data

Client -> Server : Encryption_Response_p [0x01]
- We receive verification from the client

Server -> Client : Set_Compression_p [0x03]
- Sends compression threshold value
- All further packets will be compressed now
- Optional Packet

Server -> Client : Login_Success_p [0x02]
- Send client info with confirmation

Client -> Server : Login_Acknowledge_p [0x03]
- No data
- Acks the login success
- Transition to Config state

Config:

Server -> Client : Registry_Data_p [0x07]
- Game registers biomes, dimensions, etc

Server -> Client : Feature_Flags_p [0x0C]
- Enabled game features

Server -> Client : Update_Tags_config_p [0x0D]
- Block/Item/Entity tags

Server -> Client : Clientbound_Known_Packs_p [0x0E]
- Datapacks on the server

Client -> Server : Client_Information_config_p [0x00]
- player view distance, chat settings, more...

Client -> Server : Serverbound_Known_Packs_p [0x07]
- The Clients data packs in use (?)

Server -> Finish_Config_p [0x03]
- We have finished configuration

Client -> Server : Acknowledge_Finish_Config_p [0x03]
- Client is ready for play
- Switch to play state
- Unsure if player only sends this after the packet above is sent
```

## Multithreading

### Singleton construction/destruction order
Meyer's singletons (`static X instance;` inside `getInstance()`) are destroyed in the *reverse* of their construction order at program exit. If a singleton's background thread/pool captures another singleton by raw pointer or reference (e.g. a lambda capturing `this`), that other singleton must be constructed *before* the pool starts, or a task still running/queued at shutdown can touch an already-destroyed object.

Two concrete cases in `MinecraftServer.cpp`'s `main()`:
- `World::getInstance()` is called explicitly before `WorldWorkerPool::getInstance().initialize()`, even though `World` would otherwise construct lazily on the first player connection (which happens *after* the pool exists). `WorldWorkerPool`'s queued chunk-generation tasks capture `this` as a `World*` and touch its members directly.
- `ConnectionManager::getInstance().initialize()` runs before `TickLoop::getInstance().initialize()`, since the tick thread reaches into `ConnectionManager` every tick and needs to stop first.

(`WorldWorkerPool`'s tasks also hold `shared_ptr<Connection>` captures rather than raw pointers, so they stay safe to run even after `ConnectionManager` itself is torn down — only the raw-pointer case needs the construction-order fix.)

This was reliably reproduced (2026-07-18) with an isolated test program linking the project's real compiled `.o` files: bad ordering crashed with an out-of-bounds read into the noise permutation table when a queued task ran after `World`'s destruction; correct ordering exited cleanly.

### ThreadPool shutdown semantics
`ThreadPool::~ThreadPool()` drops any *unstarted* queued tasks and only waits for already-running tasks to finish. This used to drain the entire queue before shutting down (a deliberate safety-first choice from early on), but that became a real problem once `WorldWorkerPool` had real, possibly large backlogs of chunk-gen/lighting work queued (e.g. stopping the server mid-exploration) — shutdown would hang waiting for the whole backlog instead of just in-flight work. Verified via a synthetic 400-task backlog test: close time dropped from a multi-second drain to ~374ms.

## World Generation & Lighting

### Terrain cache vs. lit cache
`World` keeps two separate chunk caches: `_terrainCache` (block data only, safe for concurrent reads since it's never mutated after generation) and `_chunkCache` (fully lit, what actually gets sent to players). `LightEngine` needs an 8-neighbor lookup of raw block data to compute a chunk's lighting correctly, and routing those lookups through `getOrGenerateTerrain` means each neighboring chunk's terrain is only ever generated once, no matter how many nearby chunks need it purely for lighting occlusion — instead of regenerating it from scratch on every lookup.

### LightEngine
Full BFS flood-fill for both sky and block light, computed once at chunk-generation time (no incremental re-lighting, since there's no block editing yet). It operates over a 3x3-chunk (48x48 block) neighborhood centered on the target chunk — this is *exactly* correct, never approximate, because the max light level (15) is less than chunk width (16), so light can never propagate further than one chunk in from the target's own borders.

Sky light propagating straight down through a transparent block doesn't attenuate, unlike every other direction (which decays by `max(opacity, 1)`) — this special case isn't spelled out by `network-protocol.md`, but has been confirmed correct through testing.

Seeding is done in two passes for performance: the first pass walks each column top-down and sets sky light values directly (15 down to the first opaque block) without touching the BFS queue at all; the second pass only *enqueues* cells that sit at an actual light/dark boundary (a horizontal neighbor not already at max light, or the buffer edge). A wide-open sky column can be a couple hundred blocks of uniform max-light air, and enqueueing every one of those cells individually made the BFS pop/check millions of neighbors that could never produce a new result.

### Chunk dispatch/delivery ordering
`UpdateLoadedChunks` (in `Play.cpp`) dispatches and delivers chunks nearest-to-center first, not in raster/`std::set` order. `Connection::sendPackets()` serializes and writes queued packets to the socket strictly in order, one at a time — so if a whole ring's worth of chunks becomes ready at once (e.g. all cache hits on reconnecting to an already-explored area), the chunk directly under the player's own feet could end up serialized behind ~200 others. The client's "waiting for chunks" gate (which holds off gravity until its own chunk arrives) would then release too late, and the player free-falls into the ground before their own chunk shows up. This was found from a real "stuck underground on reconnect" bug report and confirmed via the client's F3 debug screen, not assumed from reading the code.

### Heightmap encoding
`Chunk_Data_p` sends a `MOTION_BLOCKING` heightmap: for each of a chunk's 256 columns, the Y of the highest non-air block, packed as `(y - WORLD_MIN_Y + 1)` (0 meaning "nothing blocking in this column") at 9 bits/entry, 7 entries per long (never straddling a long boundary), column index `z*16 + x`. This is vanilla's own encoding scheme. Its practical effect today is minimal: it's mainly used client-side for weather-particle rendering (rain/snow stop at the heightmap surface instead of an assumed y=0), and there's no weather yet.

### Performance history
Two performance passes, both 2026-07-18:

1. **Algorithmic fixes** — the terrain/lit cache split above; memoizing `NoiseChunkGenerator`'s per-column `isSolid()` results instead of recomputing them up to `DIRT_DEPTH+1` times per solid block for surface layering; and the two-pass sky-light seeding above. Combined: ~58s -> ~21s for a full 441-chunk view-distance batch (measured with an isolated timing harness linked against the real `.o` files).
2. **Build flags** — turned out to be a bigger factor than any single algorithmic fix. The Makefile's default build had no `-O` flag at all (effectively `-O0`). Adding `-O2` to the default build alone (no code changes) took `NoiseChunkGenerator::generate()` from 50.8ms to 11.8ms and `LightEngine::computeLighting()` (including its 8 neighbor terrain gens) from 362.6ms to 105.1ms — about 3.5x. `debug` now explicitly forces `-O0` back on top so gdb stepping stays reliable; the old separate `make fast`/`-Ofast` target was removed since `-O2` is just the default now, not an opt-in.

## Wire format gotchas found the hard way
A couple of `Chunk_Data_p` details that don't match a naive reading of the wiki summary, both confirmed against decompiled 1.21 client bytecode:
- A single-valued (Bits Per Entry = 0) paletted container is still followed by a Data Array Length VarInt (always 0 in that case) — it's present but zero-length, not omitted entirely.
- Each chunk section has exactly one non-empty-block-count short on the wire (blocks and fluids combined), not separate block/fluid counts.

## Block breaking and placing
Breaking (`Player_Action_p`) and placing (`Use_Item_On_p`) edit the world via `World::setBlock`, which copies the target chunk, edits and relights the copy, then swaps it into the cache rather than mutating the shared original in place — `Chunk` has no internal locking, so this avoids racing other threads reading the same cached `shared_ptr`. Edits broadcast a `Block_Update_p` to every player with that chunk loaded, and ack the acting player's client (`Acknowledge_Block_Change_p`) so its local prediction doesn't flicker/revert.

`Player` has a minimal 9-slot hotbar (no full inventory, no Click Container support — Set Creative Mode Slot is a separate, narrower mechanism, see "Placeable Creative-mode blocks" below), seeded with a starting stack at join and sent via `Set_Container_Content_p`. Breaking a known block drops a visible item entity (`Spawn_Entity_p` + `Set_Entity_Metadata_p`), tracked server-side by `ItemEntityManager` so it can be picked up or despawned afterward — **except in Creative mode**, where the block is still instantly removed but no item entity spawns at all, matching vanilla (`Player_Action_p::deserialize`, `Play.cpp`, checks `player.getGamemode()` before resolving a drop; a real bug, found by the user in play-testing after `/gamemode` shipped — Creative was mining blocks instead of instantly clearing them).

Pickup is checked on every `Set_Player_Position_p`/`Set_Player_Position_and_Rotation_p`, matching vanilla's own documented behavior (`docs/network-protocol.md`'s Pickup Item section) rather than a per-tick position sweep: nearby claimed items merge into an existing hotbar stack or the first empty slot, all-or-nothing (no partial pickup across the ground/inventory boundary). `ItemEntityManager::tryClaim` makes the claim atomic so two players in range at once can't both collect the same drop. Unclaimed drops despawn after 5 minutes via `ItemDespawnSystem`, a `TickSystem` registered alongside `KeepAliveSystem`.

Dropped items also get real per-tick physics via `ItemPhysicsSystem` (gravity + single-column ground collision against `World`, re-checked every tick so breaking the block out from under a resting item makes it fall again rather than becoming permanently stuck at its stale spawn position). Pressing Q (`Player_Action_p` status 3/4, "Drop item stack"/"Drop item") spawns a tossed item using the same physics, launched from the player's facing direction (yaw/pitch) with a small upward kick — the toss speed/arc is a gameplay-feel approximation, not a wire-format detail, so it wasn't decompile-verified the way packet layouts are.

Q-drop's toss direction depends on `Player`'s stored yaw/pitch being current, which needed `Set_Player_Rotation_p` (0x1C) -- sent when the client turns without moving its feet, as opposed to `Set_Player_Position_and_Rotation_p` (0x1B) which only fires alongside a position change. It wasn't registered, so standing still and looking around before dropping used a stale facing direction.

`ItemPhysicsSystem` does *not* broadcast a position correction every tick — an early version did, and it visibly fought the client's own local physics prediction for the entity (barely noticeable for a straight vertical fall's tiny per-tick steps, but a visible forward stutter for a toss's larger horizontal ones). The client is expected to simulate gravity/drag locally from the last velocity it was told, so the server only needs to correct it (`Teleport_Entity_p` + `Set_Entity_Velocity_p`, sent together) when velocity changes by more than a small threshold (squared magnitude > 0.1) — e.g. once at toss/support-loss, once at landing — not continuously. `ItemEntityManager`'s internal position is still updated every tick regardless, since pickup-radius and despawn checks need it current even on ticks where nothing is broadcast.

Wire formats not covered by `docs/network-protocol.md`, verified through testing rather than guessed: the 1.20.5+ Slot/Data-Components format, the player inventory's 46-slot layout (hotbar at indices 36-44), and the item entity's metadata index/type for its displayed stack (index 8, type 7).

Known gaps: no reach or mining-time validation (breaking is instant), placing never replaces the clicked block itself (can't place into water/tall grass), and terrain used for lighting neighbors isn't updated by edits.

## Player entities, skins, and online-mode auth

Login is real online-mode Mojang authentication when `server.properties`' `online-mode` is true (the default): the server computes the vanilla join-hash (`computeServerHash` in `Crypto.hpp`, `SHA1(serverId + sharedSecret + publicKeyDer)` formatted as a signed hex `BigInteger`-style string) and calls `sessionserver.mojang.com`'s `hasJoined` endpoint over a small hand-rolled HTTPS client (`network/HttpsClient.hpp`, TLS via the already-linked OpenSSL, no new library dependency) before trusting the client's claimed UUID/username, replacing them with Mojang's verified values and storing the returned "textures" property (`PlayerProfileProperty` on `Player`) for skin rendering. The join-hash hex formatting was verified against the standard published SHA1("Notch")/"jeb_"/"simon" test vectors (including the negative two's-complement case), not trusted from memory alone. `online-mode=false` skips all of this and falls back to trusting the client, same as before this feature existed.

Two pre-existing bugs in `Crypto.cpp` were fixed as a prerequisite: the RSA keypair was being regenerated (not reused) on every call, which would have made the join-hash never match what was sent to the client; and the verify token was a single global instead of per-connection, letting concurrent logins clobber each other.

Other players become visible in two independent ways: a server-wide tab list (`Player_Info_Update_p`/`Player_Info_Remove_p`, synced once at join via `BroadcastPlayerJoin`) and chunk/proximity-based in-world entities (`PlayerVisibilityManager`, hooked into `Connection::deliverGeneratedChunks` — refreshing whichever connection's own loaded-chunk set just changed is enough to catch both directions of visibility, since the check itself is bidirectional). A joining player's own `Player_Info_Update_p` entry is sent back to themselves too, not just broadcast to others — the client needs it (skin variant, cape) to render its own third-person model, not only the tab list.

**Bug found via real-client testing**: the tab-list overlay (holding Tab) showed a completely empty list — just the background bar, no player rows — even though skins/entities worked fine. Root cause: `Player_Info_Update_p` only ever set the Actions bitmask's `Add Player` bit (`0x01`); the client only actually shows a player in the tab-list overlay if `Update Listed` (`0x08`, field `Listed: Boolean`, confirmed via `minecraft.wiki`'s Player Info Update table) is also sent true for them — without it, a connected player's overlay row is simply never populated, despite Add Player's own fields (name, skin) taking effect elsewhere. Fixed by sending Actions `0x01 | 0x04 | 0x08` (Add Player | Update Game Mode | Update Listed), always writing `Listed = true` (this project has no mechanism to hide a connected player from the tab list) and each player's real `getGamemode()` (bundled in at the same time since the field was already on hand at both `Entry`-construction call sites in `BroadcastPlayerJoin`, and matters now that a returning player's gamemode can be non-Survival via `PlayerDataPersistence`). `Update Latency` (`0x10`, ping) remains a known gap — this project has no round-trip Keep Alive timing to source a real number from, so a ping bar would need fabricated/placeholder data rather than a real measurement.

Wire formats not fully covered by `docs/network-protocol.md`, verified rather than guessed: the `Player Info Update` per-player field layout (UUID immediately followed by that player's own action fields, repeated per player — the vendored doc's table is ambiguous on this point), and the `minecraft:player` entity-type ID (128, from a freshly-generated vanilla data report, cross-checked against the already-known item entity-type ID). The player entity metadata index for the "Displayed Skin Parts" bitmask (hat/cape/sleeves/etc.) is index 17, type 0/Byte — confirmed against an archived minecraft.wiki revision explicitly labeled "Java Edition 1.20.2" (this field is stable across adjacent versions). The *live* wiki page shows 16 instead, but that reflects a much newer snapshot, not 1.21 — worth remembering when checking protocol details again: check that the version being shown actually matches 1.21, don't assume the "current" page/table applies.

`Acknowledge_Finish_Config_p::deserialize` also sends a player their own `Set_Player_Skin_Parts_Metadata_p` (targeting their own entity ID) right alongside `Login_Play_p`, since `PlayerVisibilityManager` never does this for a player about themselves (a player's own entity is never "spawned" to them). This mirrors the fix that got the skin *texture* showing correctly in a player's own third-person view (sending their own `Player_Info_Update_p` entry back to them, see above) — but unlike that case, this one did **not** fix the hat layer in the player's own F5 view; other players do correctly see it via the identical mechanism. Current best explanation (unconfirmed): the client's own third-person renderer may read the skin-parts bitmask from local client options directly rather than its own entity's networked metadata, for this one field specifically — an open question, not settled.

Player movement and rotation are broadcast to whoever currently has that player visible (`PlayerVisibilityManager::broadcastMovement`, called from `Set_Player_Position_p`/`Set_Player_Position_and_Rotation_p`/`Set_Player_Rotation_p` after `Player`'s own state is updated). Which packet gets sent depends on what changed: position-only movement within an 8-block delta sends `Update_Entity_Position_p`; position+rotation together sends `Update_Entity_Position_and_Rotation_p` plus `Set_Head_Rotation_p` (head yaw always mirrors body yaw here — there's no separate client-sent head-look packet to track independently); rotation-only (looking around without moving) sends `Update_Entity_Rotation_p` plus `Set_Head_Rotation_p`; any position delta exceeding 8 blocks in any axis falls back to an absolute `Teleport_Entity_p` instead, since the delta-encoded packets can't represent a jump that large. Rotation is always sent as an absolute angle, never a delta, so there's no "old rotation" to track the way position has an old/new pair for delta math.

Sneaking and sprinting are relayed the same way: `Player_Command_p` (client presses shift/sprint) updates `Player`'s state, then `PlayerVisibilityManager::broadcastPoseChange` tells every current viewer via two separate packets — `Set_Entity_Flags_Metadata_p` (index 0, bit `0x02` crouching / `0x08` sprinting) and `Set_Player_Pose_Metadata_p` (index 6, `SNEAKING`=5/`STANDING`=0) — both fields live on the base Entity class and are stable across versions, unlike the Player-specific skin-parts field. A newly-visible player's current sneak/sprint state is also sent alongside their `Spawn_Entity_p` (mirroring the skin-parts precedent), so someone already sneaking before a viewer arrives still shows up crouched immediately rather than only on the next state change.

Three real bugs along the way, all worth remembering for any future entity-metadata work: (1) Entity Flags and Pose must be sent as two separate Set Entity Metadata packets — combining both fields into one multi-entry packet made sprinting render correctly while sneaking silently didn't, even though the byte encoding was structurally valid. (2) Pose has its own dedicated metadata type ID, not the generic VarInt type (1) — its value is wire-identical to a plain VarInt either way, so this is easy to get wrong without a crash or obvious symptom: the client just silently ignores an entry whose declared type doesn't match what it expects for that index. (3) That type ID is itself version-specific, the same trap that hit the skin-parts index earlier: it's **21** for 1.21 through 1.21.7, but only **20** starting at 1.21.9 (renamed again in later snapshots) — a live wiki page defaults to showing the newest version's numbering, which gave 20 and silently failed for this project's actual 1.21 target. Neither wrong-type attempt produced any visible error, just no pose change, which is why bugs (1) and (2) each looked like the whole fix in turn before checking version-pinned data directly.

Known gaps: no player-entity metadata beyond skin parts/pose (e.g. elytra flying, riding), and a player's own F5 view doesn't show their hat layer even though other players see it correctly (see above).

## Chat and commands

Chat deliberately skips the whole 1.19+ signed-chat system (session public keys, per-message signatures, acknowledgement bitsets). `Chat_Message_p` (0x06) still has to deserialize those fields correctly since the client always sends them, but they're consumed and discarded rather than verified — this server already establishes a player's identity through online-mode Mojang verification at login, so message-level signing doesn't add anything it needs. Outgoing, plain chat and `/say` both go out as `Disguised_Chat_Message_p` (0x1E), which is what vanilla itself uses for command-originated chat precisely because it carries no signature — not `Player Chat Message`, which would require implementing signing to avoid the client showing an "unsigned"/insecure indicator.

`Disguised_Chat_Message_p` needs a `minecraft:chat_type` registry index (`"minecraft:chat"` for player chat, `"minecraft:say_command"` for `/say`), and — same trap as `dimension_type` in `Login_Play_p` — the registry's wire order isn't fixed, since `VanillaDataManager` loads it via `std::filesystem::directory_iterator`, whose iteration order isn't guaranteed. `BroadcastDisguisedChat` looks the id up by scanning `VanillaDataManager::getEntries("chat_type")` at send time rather than assuming a fixed index, mirroring the existing dimension-type lookup.

That index is **not** sent raw, though — a real bug found via live testing (client disconnect: "Failed to decode packet 'clientbound/minecraft:disguised_chat'"). Chat Type is a `Holder<ChatType>`, not a plain registry VarInt like `dimension_type` is: the wire value is `index + 1`, with `0` reserved to mean "an inline value follows" instead of a registry reference. The vendored doc's table just says "VarInt" with no mention of this, so sending the raw 0-based index made the client read index 0 ("minecraft:chat") as `0` on the wire, interpret that as "inline value," and try to parse the following Sender Name field's NBT as a `ChatType` definition instead of a text component — hence the decode failure. Not every registry-referencing field works this way (`dimension_type` in `Login_Play_p` is a plain index, confirmed by it already working); this is a per-field thing in Mojang's own codec, not a rule that generalizes, so treat each one as its own thing to verify rather than assumed.

Text components are sent as network NBT (`{"text": ...}` via `NbtTag`), not JSON strings — that's only for the Login-state Disconnect packet, which predates the switch to NBT-encoded chat components.

`Chat_Command_p` (0x04, the unsigned "Chat Command" packet) is the only command path handled; the signed variant (0x05) is never sent by a real client here since the `Commands` packet (see "Command autocomplete" below) never marks any argument as requiring signing — in fact it never encodes argument nodes at all. Commands typed in chat and commands typed at the server console now go through the same `Command`/`CommandRegistry` path via a `CommandSender` abstraction (`ConsoleCommandSender`, always permission level 4 and bypassing op checks like vanilla; `PlayerCommandSender`, whose permission comes from `OpsList` looked up fresh on every command rather than cached on `Player`, so a hand-edited `ops.json` or a mid-session `/deop` takes effect immediately). `CommandRegistry::dispatch` checks `Command::getRequiredPermission()` against the sender before calling `execute`, and sends the sender feedback itself (unknown command / no permission) so callers don't have to.

`OpsList` is a minimal `ops.json`-backed store keyed by dashless-hex UUID (not username, so a Mojang account rename doesn't silently deop someone) — level 0 removes the entry entirely rather than writing a zero. Built-in commands: `help`, `say` (permission 2), `op`/`deop`/`kick` (permission 3, and only look among currently-connected players — there's no offline username-to-UUID cache to resolve anyone else), `list`, and `stop` (permission 4, unchanged from before this feature except it now goes through the same permission-checked path as everything else).

Known gaps: no private messaging (`/tell`/`/msg`), no chat mode/coloring support (`Client_Information_config_p` still just discards the Chat Mode field), and `/op`/`/deop`/`kick` can't target an offline player by name.

## Disconnecting players with a reason

`Connection::disconnect(reason)` queues a state-appropriate Disconnect packet (`Disconnect_play_p`, 0x1D, for Play state; the existing `Disconnect_login_p` for Login state) and otherwise does nothing else — no force-closing the socket from here. That mirrors how every existing `Disconnect_login_p` use already worked before this: a real client closes its own end on receiving any Disconnect packet, which is what the *next* receive attempt on that connection's own processing thread turns into an EOF, setting `Socket`'s fd invalid and letting `Connection::isValid()` go false naturally. `ConnectionManager::processConnection`'s existing "connection no longer valid" branch (tab-list removal + `PlayerVisibilityManager::handleDisconnect`) already runs once that happens, so no new cleanup path was needed.

`kick` (a new command) is the direct use of this. `stop` now also walks every active Play connection and calls `disconnect("Server closed")` before requesting shutdown. Because actually sending a queued packet only happens on that connection's own thread the next time it runs `sendPackets()` (not synchronously from the command's own thread — `addPacket` only queues), `main()` sleeps briefly (500ms) after the shutdown loop exits, before returning, so those connections get at least one more chance to flush the kick packet before `ConnectionManager`'s static destruction tears down the connection thread pool. Config-state disconnects (kicking someone mid-loading-screen) aren't handled — no current command or shutdown path targets a connection in that state.

## Vanilla world-save (Anvil format) read/write

Terrain now persists across restarts, and an existing vanilla world save's `region/` + `level.dat` can be dropped into `<level-name>/` and loaded — even though this server's own terrain *generation* is still entirely its own (this is purely a file-format/persistence feature, not an attempt to reproduce vanilla's generator). Import targets **1.18+ saves only**: this project's `Chunk` already matches modern vanilla's Y-range (`WORLD_MIN_Y=-64`, `WORLD_HEIGHT=384`, 24 sections) and section-based chunk layout exactly, so no cross-era format migration was needed, and a save below `VanillaVersion::MIN_SUPPORTED_DATA_VERSION` (2860, confirmed against minecraft.wiki's Data version article) is rejected with a clear log message rather than an attempted migration. `VanillaVersion::CURRENT_DATA_VERSION` (3953, for this project's 1.21 target) was read directly from this project's own `server.jar`'s root `version.json` → `world_version` field, not guessed.

**NBT** (`network/Nbt.hpp`/`.cpp`) gained real *reading* for the first time — previously write-only (`serializeNetwork()`, the unnamed-root network form used for chat/registry data). Added `Byte_Array`/`Int_Array` tag types, a byte-buffer parser (`parsePayload`/`parseNamed`, threading a single `bool& ok` through recursive calls so a truncated/malformed buffer fails soft rather than crashing), and the *named-root* file-format reader/writer (`serializeFile`/`parseFile`) that `level.dat` and region-file chunks both use (as opposed to network NBT's unnamed root). Gzip wrapping (`compressGzip`/`decompressGzip` in `network/Compression.hpp`, using `inflateInit2`/`deflateInit2` with `windowBits=15+16`) is separate from the existing plain-zlib `compressData`/`decompressData` — `level.dat` is gzip, region-file chunk sectors are raw zlib (compression scheme 2), packet compression still needs plain zlib.

**`RegionFile`** (`.mca` mechanics only, no chunk semantics) is deliberately chunk-agnostic: an 8KiB header (1024-entry location table — 3-byte sector offset + 1-byte sector count — then a 1024-entry timestamp table), then per-chunk `[4-byte length][1-byte compression scheme][compressed data]`, 4096-byte sector-aligned. Overwrites in place if a rewritten chunk still fits its existing sector allocation, otherwise appends at end-of-file (no compaction/free-list — simplest correct thing, not the most space-efficient). Chunk coordinate → region file is `chunkX >> 5, chunkZ >> 5` (arithmetic right shift floors correctly for negative coordinates on its own, unlike integer division — this is why `Chunk.hpp` needed its own `floorDiv16` for `/` but this needed nothing extra for `>>`); the local within-region index is `(chunkX & 31) + (chunkZ & 31)*32`, again relying on two's-complement `&` already giving a true positive modulo for negative inputs.

A `RegionFile` deliberately does **not** create its `.mca` file just from being opened/queried — only `writeChunk` creates one lazily, on the first real save. The first version did create it eagerly, and real testing immediately surfaced why that's wrong: `WorldPersistence` opens a `RegionFile` for every region a chunk lookup touches, including `LightEngine`'s neighbor-chunk queries near a region boundary, which would otherwise litter `region/` with empty 8KiB files for regions nothing was ever actually saved to.

**`ChunkNbtCodec`** translates a chunk-root NBT compound ↔ this project's `Chunk`, both directions. Reads the `sections` list (`Y` can be negative — `WORLD_MIN_Y=-64` means section `Y=-4` is the lowest), each section's `block_states.palette` (name+properties compounds, translated via a new small `BlockNames` table — see below) and bit-packed `data` `LongArray` (`bitsPerEntry = max(4, ceil(log2(paletteSize)))`, non-spanning packing where entries never cross a `long` boundary — the scheme in use since 1.16). A palette of size ≤1 has **no** `data` array at all on disk (vanilla's own optimization for a uniform section, e.g. all-air) and must be filled uniformly rather than treated as an error. Only `Status == "minecraft:full"` chunks are imported faithfully; anything else (partially-generated edge-of-world chunks) falls back to this project's own generator rather than attempting to emulate vanilla's multi-stage generation pipeline. Biome: vanilla's per-4×4×4-cell storage is coarser than `Chunk`'s single whole-column `_biomeId` (which no generator had ever actually set before this — `Chunk::setBiomeId` is new), so the first section's first palette entry is taken as representative; the biome name↔index translation reuses the same "look it up by id, since registry send order isn't fixed" convention already used for `dimension_type`/`chat_type` elsewhere, against `VanillaDataManager`'s `worldgen/biome` entries.

**Light data is never read or written at all**, on either side. Vanilla's own format already supports this via `isLightOn=false` ("recompute me"), and this project already always recomputes lighting fresh via `LightEngine::computeLighting` for every chunk regardless of origin — so persisting on-disk light nibble arrays would be pure unused complexity. Encode always writes `isLightOn=0`; decode ignores any present `BlockLight`/`SkyLight`. Entities and block-entities (chests, furnaces, mobs, dropped items) are likewise never read or written — this project has no block-entity or mob concept yet, and none of its known blocks are containers, so this is a natural non-goal rather than a real design fork.

**`BlockNames`** (new, small hardcoded table, same style as `ItemBlockMapping`) covers exactly the 4 block-state IDs this project knows today. An unknown/unsupported name on import (overwhelmingly likely against a real vanilla world, given how small this palette still is) substitutes stone — solid and safe, no fall-through hazards — logged once per *distinct* unknown name rather than once per block instance, so import fidelity loss is visible without spamming the console. `minecraft:air`/an empty name map to air specifically, so empty space doesn't turn solid.

**`level.dat`** (`LevelDat`) is a minimal field set (`DataVersion`, `LevelName`, `RandomSeed`, `SpawnX/Y/Z`, `SpawnAngle`, `GameType`, `Difficulty`, `hardcore`, `Time`/`DayTime`), not byte-perfect vanilla parity — game rules, world border, and datapacks are simply never read or written. Writes go through write-to-temp-then-rename for crash safety (a half-written `level.dat` from a mid-save crash would otherwise brick the world). `Properties::level_seed`/`level_type` are read for the very first time by this feature (previously dead fields — `World`'s seed/flatness were hardcoded defaults before this): an existing world's `level.dat` is authoritative for seed/spawn when present (matches vanilla — `level-seed` in `server.properties` only matters for a brand-new world), while `level_type` always governs which generator fills in anything not already covered by a save, independent of whatever generator the save's original world actually used. A text `level-seed` is hashed the same way real vanilla does (verified against minecraft.wiki, not assumed): a string that parses fully as a signed integer is used directly; anything else is hashed via Java's `String.hashCode()` (32-bit signed, computed with unsigned wraparound arithmetic to avoid C++ signed-overflow UB, then sign-extended into the 64-bit seed field); an empty string gets a genuinely random seed, same as vanilla does for a brand-new world with no configured seed.

**Wiring into `World`**: `getOrGenerateTerrain` checks `WorldPersistence::tryLoadChunk` before falling back to `_generator->generate(...)`. Dirty tracking uses two small coordinate sets guarded by the existing `_chunkCacheMutex` (no new lock-ordering concern): `_dirtyChunks` (needs saving) and `_cleanChunks` (just loaded from disk, unmodified — consumed the first time `cacheAsLit` runs for that coordinate, so a *later* edit via `setBlock` still marks it dirty normally). This avoids needlessly rewriting an untouched imported chunk every autosave cycle without needing to change `cacheAsLit`'s signature or touch unrelated call sites (`LightEngine`'s neighbor queries, the constructor's spawn-chunk priming). `AutosaveSystem` (a new `TickSystem`, registered in `TickLoop::initialize()` alongside the existing systems) saves every dirty chunk plus `level.dat` every 5 minutes (`tickCount % (5*60*20)`, matching vanilla's default cadence); the same `AutosaveSystem::saveNow()` also runs once more from `main()` right after the shutdown loop exits, and a new `/save-all` command (permission 4, matching vanilla) triggers it on demand.

`WorldPersistence::initialize(worldDir)` is called explicitly by `World`'s constructor immediately after resolving `_worldDir`, rather than `WorldPersistence` reading `World::getInstance().getWorldDir()` itself — `World`'s own constructor calls `getOrGenerateTerrain` for its spawn-chunk priming, which reaches into `WorldPersistence`, and `World::getInstance()` would then be called reentrantly on the same thread while its Meyer's-singleton static local is still under construction (undefined behavior — libstdc++ throws on detecting this). Caught during implementation, not by hitting the crash first.

Verified end-to-end with a real (headless) server run rather than just the offline codec tests: `/save-all` correctly wrote exactly one region file plus `level.dat`; restarting against the same world directory logged the seed loaded from the existing `level.dat` (a text seed, confirming the hashing above); a third run left the region file's mtime untouched (the loaded chunk was correctly marked clean) while `level.dat` still updated. A real vanilla save import and the pre-1.18 rejection path (both otherwise covered by scratch-tested `LevelDat`/`ChunkNbtCodec` version checks in isolation) still need a real client and a real save file to confirm fully live.

Known gaps: no entity/block-entity persistence, no Nether/End (this project has one dimension), no byte-perfect `level.dat` parity, no pre-1.18 import (rejected outright), and no non-blocking (fork-based) save yet — this is the synchronous version, deliberately sequenced first.

## Chunk unloading (RAM eviction)

`World`'s `_chunkCache`/`_terrainCache` never shrank before this — every chunk ever touched stayed resident for the life of the process, so RAM climbed monotonically as players explored. Now made safe by the persistence work above (an evicted chunk can always be regenerated or reloaded from disk on demand, and a dirty one is saved first).

A reverse index — `_chunkViewerCounts` (coordinate → number of players currently holding it loaded) and `_zeroViewerSince` (coordinate → when its count last hit zero), both new `World` fields guarded by the existing `_chunkCacheMutex` — is maintained by `World::chunkViewerAdded`/`chunkViewerRemoved`, called from the three places a player's own loaded-chunk set already changes: `Connection::deliverGeneratedChunks()` (add, alongside `markChunkLoaded`), `UpdateLoadedChunks`'s unload loop in `Play.cpp` (remove, alongside `markChunkUnloaded`), and a new hook in `ConnectionManager::processConnection`'s Play-state disconnect branch (remove for every coordinate in `getLoadedChunks()` — nothing decremented anything on disconnect before this).

Eviction uses a **grace period + periodic sweep**, not immediate eviction the instant a chunk's last viewer leaves — avoids repeatedly evicting/reloading a chunk a player is oscillating right at their view-distance edge of. A new `ChunkUnloadSystem` (`TickSystem`, registered in `TickLoop::initialize()` like the other systems) sweeps every 5 seconds (`SWEEP_INTERVAL_TICKS=100`) and evicts anything past `GRACE_PERIOD_SECONDS` (a single named, deliberately easy-to-tweak constant, currently 15.0) via `World::evictStaleChunks`.

The actual per-chunk eviction (`World::evictChunk`) re-checks the viewer count in the *same* lock acquisition as the `_chunkCache`/`_dirtyChunks`/`_cleanChunks` erase, rather than a snapshot-then-erase-later shape — closes a narrow race where a viewer returning (or a stray in-flight edit) between "the sweep decided to evict" and "the erase actually happens" could otherwise be silently lost. Only chunks in `_dirtyChunks` get saved (via `WorldPersistence::saveChunk`, called unlocked, after the cache lock is released — disk I/O is never done while holding `_chunkCacheMutex`, matching `setBlock`/`takeDirtyChunksSnapshot`'s existing convention). `_terrainCache` is swept too (`World::evictStaleTerrainCache`) — same memory cost as a lit chunk and unbounded before this — but needs no grace period or save step since these entries are never edited/dirty-tracked; anything not currently in `_chunkViewerCounts` is simply dropped, regenerated cheaply if needed again.

Known, accepted gap: the constructor's non-flat-generator spawn-chunk prime at (0,0) (`cacheAsLit(0, 0, spawnChunk)` in `World::World()`) never goes through `chunkViewerAdded`, so it never enters the reverse index and is exempt from eviction until a real player loads (0,0) through the normal view flow. One fixed chunk, not a growing leak.

Verified offline via a scratch program driving `World`'s viewer-count/eviction methods directly against a flat-generator world (fast, deterministic): a chunk with zero viewers isn't evicted before its grace period elapses, is evicted (and its dirty data correctly saved+reloadable) once it does, a viewer returning before the grace period elapses aborts the eviction, and `_terrainCache` correctly evicts unviewed entries while keeping viewed ones (checked via object-identity: a cache hit returns the same `shared_ptr`, a re-generation after eviction returns a different one). A real-client check (walk out of view distance, wait past grace+sweep, walk back, confirm correct reload, watch RSS stay flat during exploration) still needs the user's own client.

## Per-player persistence (playerdata/&lt;uuid&gt;.dat)

Only world-level state (chunks, spawn point) was persisted before this — a player's position, rotation, gamemode, and hotbar were purely in-memory `Player` fields, reset to the world spawn point (and a hardcoded starting hotbar stack) on every single join, even a rejoin by the same player. `PlayerDataPersistence` (`include/PlayerDataPersistence.hpp`/`src/PlayerDataPersistence.cpp`) closes that gap for the subset of `Player` state that actually has something to persist — health, XP, achievements, and a full 36-slot inventory don't exist as `Player` state at all, so there's nothing to save for them (not a scope cut, a non-goal). Deliberately free functions (`tryLoad`/`save`), not a singleton — matches `LevelDat`'s shape rather than `WorldPersistence`'s, since there's no open-handle state to cache here (each save/load is one small, one-shot file operation).

File location matches real vanilla: `<worldDir>/playerdata/<uuid>.dat`, dashed (`8-4-4-4-12`) to match vanilla's own filename convention — the only existing UUID→string helper, `uuidToHexString` (`network/PacketUtils.hpp`, used today only as an `OpsList` lookup key), is dashless, so a small local helper in `PlayerDataPersistence.cpp` derives the dashed form from it rather than reimplementing hex conversion. NBT shape is loosely vanilla-named for discoverability with real NBT tools, not byte-perfect parity (the same stance already taken for `level.dat`): `DataVersion`, `Pos` (List<Double>×3), `Rotation` (List<Float>: yaw, pitch), `playerGameType`, `SelectedItemSlot`, and `Inventory` (List<Compound>, one entry per *occupied* hotbar slot only, `{Slot, id, Count}` — hotbar slots 0-8 are exactly vanilla's own hotbar slot numbering within a full inventory, so this lines up naturally). Item names reuse the existing two-hop conversion (`blockStateIdToName(itemIdToBlockStateId(itemId))` and its inverse) already used for dropped-item conversion in `Play.cpp` — no dedicated item-name registry exists, but every item this project has today is one of the 3 known block-items, so the round trip is exact. Writes use the same write-to-temp-then-rename pattern as `LevelDat::save`.

This is the first place in the codebase where the *same* file could plausibly be written from two different threads close together: a connection-pool thread's disconnect-triggered save can race the tick thread's periodic autosave sweep for that same still-connected player. A single file-scope `std::mutex` in `PlayerDataPersistence.cpp` serializes every `save()` call — simplest fix given how infrequent and cheap these writes are; a per-UUID lock map would be solving a problem that doesn't exist at this scale. Worst case on a near-simultaneous race is one redundant, still-correct write, not corruption.

Wired at three points: (1) **join** — `Acknowledge_Finish_Config_p::deserialize` (`Config.cpp`) tries `PlayerDataPersistence::tryLoad` before falling back to the world spawn point/starting stack, and now also calls `player.setGamemode(...)` for the first time anywhere in the codebase (the method already existed on `Player`; nothing had ever called it, so gamemode was always the compile-time Survival default — this is a strict improvement, not a behavior change for anything that depended on it). (2) **disconnect** — `ConnectionManager::processConnection`'s Play-state branch saves `conn->getPlayer()` right alongside the chunk-viewer cleanup added for chunk unloading; `Player` is a plain value member of `Connection`, so it's still fully populated with final state at that point, not reset or destroyed first. (3) **periodic/shutdown** — `AutosaveSystem::saveNow()` now also iterates every active Play-state connection and saves it, after the existing chunk/`level.dat` save; this one change covers both the 5-minute cadence and the shutdown-time call already made from `main()`, without depending on kick-packet flush timing.

Verified offline via a scratch program: constructed a `Player`, set position/rotation/gamemode/hotbar (including an untouched slot)/selected-slot, saved, confirmed the file landed at the exact expected dashed-UUID path, `tryLoad`ed it back and confirmed every field round-trips exactly (including that the untouched hotbar slot stays empty), and confirmed `tryLoad` on an unknown UUID returns `nullopt` rather than an error. A real end-to-end check (join, move, change held slot, place/break to change the hotbar, disconnect, rejoin, confirm everything is restored instead of resetting to spawn) still needs the user's own client.

**Bug found via that real-client check, not by inspection**: a returning player would rejoin embedded a few blocks underground. Root cause was two latent bugs in `Acknowledge_Finish_Config_p::deserialize` (`Config.cpp`), both invisible until a player could resume anywhere other than spawn:
1. `UpdateLoadedChunks(cont, threshold, player, 0, 0)` hardcoded chunk (0,0) as the initial view center regardless of the player's actual (possibly restored) position — harmless only because every prior join really was at (0,0).
2. `Synchronize_Player_Position_p` (the teleport) was queued before the player's own standing chunk was guaranteed to be ready. At spawn, chunk (0,0) is synchronously pre-warmed in `World`'s constructor at startup, so this was always instant. A restored position elsewhere is a cache miss — generated/loaded async on `WorldWorkerPool`, taking measurably longer — so the client was teleported into an unloaded column and free-fell until real terrain arrived a moment later, landing embedded below the surface.

Fixed by computing the player's real home chunk (`std::floor(x / 16.0)`, `std::floor(z / 16.0)`, matching the existing conversion used by `Set_Player_Position_p` et al. in `Play.cpp`) right after position is resolved, passing it to `UpdateLoadedChunks` instead of the hardcoded `(0, 0)`, and calling a new `World::ensureChunkLoaded(chunkX, chunkZ)` — a synchronous counterpart to `getChunkAsync` that generates/loads + lights + caches a chunk directly on the calling thread instead of dispatching to `WorldWorkerPool` — before `Synchronize_Player_Position_p` is even constructed. This guarantees the standing chunk's `Chunk_Data_p` goes out in the very same packet flush as the teleport (via the existing `getChunkAsync` cache-hit fast path inside `UpdateLoadedChunks`'s dispatch loop), closing the race for the one chunk collision actually depends on; neighboring chunks still stream in asynchronously as before.

## Command autocomplete

Registered commands only ever existed server-side before this — a player typing `/` in chat got no client-side autocomplete, since the client was never told what commands exist. `Commands_p` (`network/packets/Play.hpp`/`.cpp`, packet `0x11`, Play/clientbound) fixes that, sent once at join (`Acknowledge_Finish_Config_p::deserialize`, `Config.cpp`) alongside the other static per-connection join packets (`Set_Default_Spawn_Position_p`, `Synchronize_Player_Position_p`, `Set_Container_Content_p`).

**Deliberate scope line**: the real Commands packet's wire format is a flat array of nodes (root/literal/argument), each with a flags byte and a children-index array — but *argument* nodes additionally need a Brigadier parser-ID (a registry that has changed across versions, confirmed via `minecraft.wiki`'s Command Data page) plus per-parser property encoding, neither of which this project can verify from memory. Since every one of this project's 8 commands is either a bare literal (`help`, `list`, `stop`, `save-all`) or a literal followed by free-text args this project never validates or type-checks today (`Command`'s interface has no argument-arity/type metadata at all — args are just a raw `vector<string>` split on whitespace), the scope here is **every command is advertised as a bare executable literal node**, root → one child per command, no argument children at all. This fully satisfies "commands autofill in chat" (typing `/ki` suggests `/kick`) while never touching the version-fragile parser-ID encoding — a deliberate line, not a shortcut around something the feature actually needed. The root/literal encoding itself (flags bitmask: `0x03` node type, `0x04` executable, `0x08` has-redirect, `0x10` has-suggestions-type) is stable and well-documented, unlike the parser-ID table.

`Commands_p`'s constructor filters `CommandRegistry::getInstance().getCommandNames()` down to whatever the joining player's permission level actually allows (`permissionLevel >= command->getRequiredPermission()`), mirroring `HelpCommand`'s own existing filter — same convention, not a new design decision. The permission lookup itself (`OpsList::getInstance().getOpLevel(uuidToHexString(player.getUUID()))`) is the exact same one `PlayerCommandSender::getPermissionLevel()` already does, reused directly in `Config.cpp` rather than constructing a full `CommandSender` just to read one int.

**Update, found via real-client testing**: the above gap (autocomplete going stale after a mid-session `/op`/`/deop`) was real and immediately noticeable — a freshly-opped player's new commands didn't autofill until reconnecting. Fixed by having `OpCommand`/`DeopCommand` (`src/commands/OpCommand.cpp`/`DeopCommand.cpp`) resend `Commands_p` to the target connection right after `OpsList::setOpLevel`, using the new permission level — both commands already hold the target's `Connection` at that point (they only ever operate on currently-connected players, per the note above), so this is a one-line addition per command, no new lookup or hook needed. `/op` resends with `Properties::getProperties().op_permission_level`; `/deop` resends with `0`. Dispatch-time enforcement (`CommandRegistry::dispatch`) was never affected either way — this only closes the client-side autocomplete lag.

Verified offline via a scratch program: constructed `Commands_p` at permission levels 0/2/4, hand-decoded the resulting bytes (packet ID, node count, root node's flags/children, each literal node's flags/children/name, trailing root index), and confirmed the exact expected command subset survives filtering at each level, in `CommandRegistry`'s registration order. A real client check (join, type `/`, confirm autocomplete appears; confirm a lower-permission account's list correctly omits commands it can't run) still needs the user's own client.

**Extension: per-command literal argument suggestions** (e.g. `/gamemode` autocompleting `survival`/`creative`/`adventure`/`spectator`). New `Command::getArgumentSuggestions()` (`Command.hpp`, default `{}`, only `GamemodeCommand` overrides it) stays inside the same deliberate scope line above — it's still purely literal nodes, not real Brigadier argument/parser-ID encoding, so it doesn't reopen the version-fragile area at all. `Commands_p` was restructured from a flat `vector<string>` into an explicit `vector<Node>` (`{flags, children, name}`) built once in the constructor: a command with no suggestions is still exactly the original bare executable leaf; a command *with* suggestions gets a non-executable node (a bare `/gamemode` alone isn't a complete command) whose children are one executable literal per suggestion string. `serialize()` just walks whatever `_nodes` the constructor built, unchanged in spirit from before. One correctness note from writing this: `_nodes.push_back(...)` inside the building loop can reallocate the vector, so root's children are accumulated into a local `vector<int>` and assigned to `_nodes[0].children` only once, at the end, rather than held as a live reference across the loop (a real, if narrow, use-after-reallocation risk caught during implementation, not by hitting a crash).

Verified offline via a scratch program: hand-decoded a full permission-4 `Commands_p`, located the `gamemode` node among root's children, confirmed it's literal-but-**not**-executable (`0x01`, no `0x04` bit) with exactly 4 children in order (`survival`/`creative`/`adventure`/`spectator`), each of those literal-**and**-executable (`0x05`) with no children of their own; also confirmed an unrelated bare command (`help`) is completely unaffected by the change, still a plain executable leaf.

## Day/night cycle

`World` had zero notion of elapsed time before this — `LevelData` (`LevelDat.hpp`) already had an `Int64 dayTime` field, and `LevelDat::tryLoad` already read a saved world's `DayTime` NBT tag back into it, but `World::buildLevelData()` hardcoded `data.dayTime = 0` on save and the constructor never read `loaded->dayTime` back out on load — a stub waiting to be wired up, not new plumbing. A new `std::atomic<Int64> _dayTime` field on `World` (atomic since the tick thread is the sole writer while connection threads read it for join packets/broadcasts/`buildLevelData()` — no mutex needed for a single scalar with no cross-field invariant) closes both ends: the constructor now sets `_dayTime = loaded->dayTime` inside the existing `if (loaded)` branch, and `buildLevelData()` now sends `getDayTime()` instead of a hardcoded `0` — which means day time persists across restarts for free, riding on `AutosaveSystem`'s already-existing periodic/shutdown `level.dat` save with no new save hook.

New `Update_Time_p` (`network/packets/Play.hpp`/`.cpp`, packet `0x64`, Play/clientbound, confirmed via `docs/network-protocol.md:2924-2935`) carries two `Long` fields, **World Age** then **Time of day** — vanilla lets these diverge via `/time set`, but since this project has no such command (and `LevelDat.cpp` already writes `DayTime`/`Time` as the same value, an existing simplification this feature just keeps), the packet takes one `Int64` and sends it for both fields. New `DayNightSystem` (`systems/DayNightSystem.hpp`/`.cpp`, a `TickSystem` registered in `TickLoop::initialize()` like the others) calls `World::advanceDayTime()` every tick (1 tick = 1 unit of day time, matching the vendored doc's "20 per second" cadence at 20 TPS) and broadcasts `Update_Time_p` to every active Play-state connection once a second (`BROADCAST_INTERVAL_TICKS=20` — clients interpolate the sun/moon smoothly between updates on their own, no need to send every tick). `Acknowledge_Finish_Config_p::deserialize` (`Config.cpp`) also sends one immediately at join, so a joining player's sky is correct right away instead of waiting up to a second for the first periodic broadcast.

Deliberately out of scope: no gamerule system exists anywhere in this project (`Properties.hpp` is a flat `server.properties`-derived field list; `LevelData`'s own header comment already states game rules are simply never read or written), so there's no `doDaylightCycle`-equivalent freeze toggle — the negative-time-of-day "frozen sun" convention the wire format supports (documented right in the vendored packet table) is never used; day time always advances continuously, matching vanilla's own default state. Also no `/time set` command, per the World Age/Time of day conflation above.

Verified offline via a scratch program (in two variants — an in-process run, plus a two-process write/read pair against the same world directory to confirm real cross-restart persistence, mirroring the world-save feature's own headless-run checkpoint): a brand-new world starts at day time `0`; `advanceDayTime()` increments by exactly 1 per call; `Update_Time_p`'s serialized bytes hand-decode to packet ID `0x64` with both Long fields correctly matching the input value; `buildLevelData()`/`LevelDat::save`/`tryLoad` round-trip a nonzero day time exactly; and a second process pointed at the same world directory correctly restores the exact day time (`12345`) the first process had advanced to and saved. A real client check (join, watch the sky advance in real time; restart and rejoin, confirming time continues rather than resetting to dawn) still needs the user's own client.

## Server List Ping player count

`Status_Response_p`'s (`network/packets/Status.cpp`) player-count and sample fields were both literal `TODO` stubs — `"online"` hardcoded to `0`, `"sample"` a permanently empty array — so a real Minecraft client's multiplayer server list always showed 0 players regardless of who was actually connected. Fixed by counting real Play-state connections the same way `AutosaveSystem`/`DayNightSystem` already do (`ConnectionManager::getInstance().getActiveConnections()`, filtered to `ConnectionState::Play` — Status/Login-state connections are still mid-handshake, not actual players) and populating the JSON `sample` array with each one's `{"name", "id"}` (the vanilla-standard dashed UUID string).

That dashed-UUID formatting already existed, privately, inside `PlayerDataPersistence.cpp` (added for vanilla-matching `playerdata/<uuid>.dat` filenames) — needing the identical format a second time here was the trigger to hoist it into a real shared utility, `uuidToDashedHexString` (`network/PacketUtils.hpp`/`.cpp`), removing the local duplicate rather than copy-pasting it a second time.

No cap on the sample list's size (real vanilla's is configurable via a `player-sample-count`-style setting this project doesn't have) — every currently-connected player is included, which is fine at this project's scale; a cap can be added later if it ever matters.

Verified offline via a scratch program: `uuidToDashedHexString` produces the correctly-dashed form; `Status_Response_p` with zero real connections active serializes valid JSON with `players.max` reflecting `Properties::max_players`, `players.online` at `0`, and `players.sample` as an empty array (there's no way to fake a real `Connection`/`Player` offline, so a nonzero online count needs a real client actually connected — join and check the server list / run `/list` to confirm the count updates).

## `/gamemode` and `/time set` commands

Two testing-focused commands (permission level 2, matching vanilla's own default op-level and this project's `/say`). `TimeCommand` only implements the `set` subcommand (`day`=1000/`noon`=6000/`night`=13000/`midnight`=18000, confirmed against `minecraft.wiki`'s `/time` page, or a plain non-negative tick count) — no `/time add`/`/time query`, since testing was the only ask. `GamemodeCommand` accepts `survival`/`creative`/`adventure`/`spectator` or `0`-`3`, targeting `args[1]` if given else the sender's own name (there's no `CommandSender`→`Player` shortcut — `CommandSender` deliberately exposes none — so self-targeting still goes through the same "match a connected player by username" loop `OpCommand`/`KickCommand` already use).

Building `/gamemode` surfaced two real gaps in what a *live* gamemode change actually needs beyond `Player::setGamemode()` (which existed already but, until now, was only ever called once, at join):

- **`Game_Event_p`** (existing, packet `0x22`, generic `(event byte, float value)`) had exactly one call site, event `13` ("start waiting for level chunks"). Event `3` ("Change game mode", value = new gamemode as a float, confirmed via `docs/network-protocol.md:1806-1811`) is what actually tells an already-connected client's UI/hotbar to switch — no packet-class changes needed, just a second call site.
- **A clientbound Player Abilities packet didn't exist in this codebase at all** (confirmed via full-tree grep) — so even with the Game Event sent, a client switched to Creative would show the right hotbar but still couldn't actually fly or instant-break, since those are gated behind flags only this packet grants. New `Player_Abilities_p` (`network/packets/Play.hpp`/`.cpp`, packet `0x38`) carries a Flags byte (`Invulnerable=0x01`, `Flying=0x02`, `Allow Flying=0x04`, `Instant Break=0x08`, per `docs/network-protocol.md:2292-2310`) plus flying-speed/FOV-modifier floats. `0x38` is already used by an existing *serverbound* packet (`Use_Item_On_p`) — confirmed harmless, since this codebase already treats clientbound/serverbound IDs as independent namespaces (`0x2F` is likewise already shared between an outgoing and an incoming packet today). A new free helper, `abilitiesFlagsForGamemode(int gamemode)` (declared/defined alongside `BroadcastPlayerJoin` in `Play.hpp`/`.cpp`), maps gamemode → flags: Creative grants `Invulnerable | Allow Flying | Instant Break` (not `Flying` itself — matches vanilla's real behavior of only granting the *ability*, letting the client's own double-tap-space gesture toggle actual flight); Spectator grants `Invulnerable | Flying | Allow Flying` directly, since spectator always flies/noclips; Survival/Adventure get no flags. Sent both at join (using whatever gamemode `PlayerDataPersistence`/the default branch resolved — nothing sent this before, so a player rejoining already in Creative previously got no fly grant at all) and by `/gamemode` (target connection only — abilities are self-only, never broadcast).

No serverbound Player Abilities handling was added, deliberately — confirmed safe to skip: `Packet_Registry::fetchIncomingPacket` bounds-checks and returns `nullptr` for any unregistered ID, and `Connection::deserializePacket` already silently ignores a `nullptr` handler. This project has no anti-cheat/flight-speed validation to keep in sync with a client's fly-toggle acknowledgment anyway, so the client's local flight works fine once `Allow Flying` is granted, with or without the server ever hearing about a toggle.

`/gamemode` also rebroadcasts a `Player_Info_Update_p` entry for the target (with the new gamemode) to every connected player, including the target — `Player_Info_Update_p` is otherwise only ever sent once, at join, so without this the target's tab-list icon would go stale immediately, the same category of gap the `/op`/`/deop` `Commands_p` resend fix closed for command autocomplete.

`World::setDayTime(Int64)` (new, alongside the existing `advanceDayTime()`/`getDayTime()`) is a plain atomic store — `TimeCommand` calls it then immediately broadcasts `Update_Time_p` to every active Play connection (the exact loop shape `DayNightSystem`'s own periodic broadcast already uses), so the sky changes right away instead of waiting up to a second for the next tick. **Accepted, documented quirk**: `_dayTime` still conflates "World Age" and "Time of day" into one value (a simplification from the day/night cycle feature, made when no `/time set` existed to ever need them to diverge). `/time set` will now also jump the "World Age" field `Update_Time_p` sends — a real deviation from vanilla, where World Age never changes via any command — but nothing in this project reads or displays World Age for anything else, so it's cosmetically harmless. Splitting them into two real persisted fields was considered and rejected as unneeded complexity for a value nothing depends on.

Verified offline via a scratch program: `abilitiesFlagsForGamemode` returns the correct flags for all 4 gamemodes; `Player_Abilities_p` serializes to packet ID `0x38` with the right flags/flying-speed/FOV-modifier bytes; `World::setDayTime`/`getDayTime` round-trip and are correctly reflected in a freshly-built `Update_Time_p`'s bytes. Command dispatch/target-resolution logic (string parsing, matching a connected player by name) has no offline-scratch precedent in this project — `OpCommand`/`KickCommand` were verified live, since they need a real `Connection` — same approach here: a real client check (switch self to Creative and confirm flight/instant-break actually work, not just the hotbar changing; `/gamemode creative <other player>` with a second client connected, confirming their tab-list icon updates for everyone; `/time set noon`/`/time set midnight` confirming the sky changes immediately) still needs the user's own client.

## Placeable Creative-mode blocks (real Creative inventory)

This project only knew 4 blocks (air, stone, dirt, grass_block), and the only way to get an item into your hotbar was the hardcoded starting stone stack at join. Two things were needed to let the user place a broad range of blocks via the *real* Creative inventory screen: more block data, and a way for the client's own creative-tab interactions to reach the server.

**Block data is real, not guessed** — generated directly by running `server.jar --reports` (needs a newer JRE than this project's default; `/usr/lib/jvm/java-21-openjdk` was used locally since the bundled data-generator main class requires a newer class file version than Java 17) against this project's own `bin/server.jar`, the same source already used for `DataVersion`/entity-type IDs elsewhere. `blocks.json` gives every block's state(s) and whether it has any properties at all; `registries.json`'s `minecraft:item` entries give each item's real network protocol ID. Filtering to blocks with **exactly one state and zero properties** — no directional/waterlogged/multi-part/axis/lit variance, so stairs, slabs, doors, logs, and anything else with real per-instance state stay out of scope, a deliberate line for the same reason the Commands packet's argument nodes stopped short of full parser-ID encoding — and cross-referencing against a same-named item yielded **179 additional real, verified blocks**: full plank sets (all 11 wood types), wool/concrete/terracotta/stained-glass (all 16 colors each), common stone variants, all 7 ore types (both stone and deepslate versions) plus their raw/refined blocks, nether/end blocks, and assorted others (glass, ice, prismarine, quartz, purpur, clay, sand, gravel, etc.).

**New `include/BlockTable.hpp`/`src/BlockTable.cpp`** consolidates this into one canonical `{blockStateId, itemId, name}` table (182 rows: the original 3 plus the 179 new ones) — the previous approach (`BlockNames.cpp`'s name↔ID if-chain and `ItemBlockMapping.cpp`'s itemId↔blockStateId switch, hand-maintained separately) would have tripled ~180 entries' worth of data across two files that had to stay in sync by hand. `BlockNames`/`ItemBlockMapping` keep their exact existing public signatures — nothing else in the codebase needed to change — and now just linear-scan the shared table internally (182 entries is nowhere near large enough to need a hash map for a not-hot-path lookup). `grass_block`'s `snowy` special case stays its own branch rather than a table lookup (**a real bug caught by the offline scratch test, not by inspection**: since `grass_block` *is* a table entry representing its non-snowy state, letting `snowy=true` fall through to the generic table scan made it incorrectly match anyway instead of falling through to the unknown-substitutes-stone path like before — fixed by branching on the name *before* the table scan, not inside it).

**New `Set_Creative_Mode_Slot_p`** (`network/packets/Play.hpp`/`.cpp`, packet `0x32`, Play/serverbound) is the client's own mechanism for "pick an item from the creative tabs (entirely client-side, the server never sends their contents) and place it in a slot" — structurally separate from Click Container (`0x10`, still out of scope), confirmed via the vendored protocol doc's own note that creative-menu middle-click/drag interactions use this packet instead. Fields: `Slot` (Short, this project's existing container-slot convention: 36-44 is the hotbar) and `Clicked Item` (the same Slot *format* this project already serializes via `packSlot`, just decoded in reverse for the first time — a new `unpackSlot` helper). A nonzero component-add/remove count (real items with enchantments, custom names, etc. — nothing this project's item set ever produces) is treated as "unsupported, no-op" rather than parsed, safe because each packet's own outer length prefix already demarcates its exact byte range before dispatch. Gated on `player.getGamemode() == Creative` — a cheap one-line check, since a non-Creative sender could otherwise exploit this for free items. `Slot == -1` (dropped outside the inventory window) spawns a real dropped item entity, reusing the exact same `ItemEntityManager` machinery the Q-drop path already uses; `36-44` sets/clears the corresponding hotbar slot; anything else (crafting grid, armor, offhand) isn't modeled and is ignored.

**Two related Creative-mode bugs fixed alongside this**, same family as the already-fixed "Creative mining drops an item" bug: `Use_Item_On_p` was unconditionally consuming one from the held stack on every placement, regardless of gamemode — Creative now leaves the slot untouched (infinite blocks, matching vanilla), gated by the same `player.getGamemode()` check.

Known, confirmed-unaffected gap: no "tell other players what's in my hand" mechanism exists anywhere in this project (no `Set_Entity_Equipment`-equivalent packet) — every hotbar mutation, old and new, only ever updates the acting player's own client view. Not something this feature needed to solve.

Verified offline via a scratch program: the block table has exactly 182 entries; the original 3 blocks (including grass_block's snowy special case, both true and false) round-trip identically to before this refactor; a sample of new blocks across several categories (wool, ore, stone variant, glass, planks, redstone) round-trip name↔stateId and itemId↔stateId exactly against the generated data; unknown names/IDs still fall back the same way they always did. `Set_Creative_Mode_Slot_p::deserialize` and the `Use_Item_On_p` gamemode gate need a real `Connection`/`Player` to exercise end-to-end (no offline precedent for live packet handlers in this project, same as command dispatch) — verified via real client instead: open the Creative inventory, place several new block types across categories, confirm they render and persist correctly across a chunk save/reload; confirm placing no longer depletes the stack in Creative; confirm dragging an item out of the inventory window spawns a real, pickup-able item; confirm breaking a newly-added block in Survival still drops it correctly.

## A single bad packet could crash the whole server

Found via a real crash (three, actually, across one afternoon of testing) — `coredumpctl` showed the same signature (`__cxa_throw` → uncaught → `std::terminate` → `abort`) in three completely unrelated-looking places (`Handshake_p::deserialize`, `PlayerDataPersistence::save`, an `NbtTag` vector append), which was the tell that this was one systemic bug, not three separate ones.

**Root cause**: `Socket::receivePacket()` (`network/Socket.cpp`) called `recv()` exactly once and returned whatever came back, without checking it actually got the full declared packet length. TCP doesn't guarantee a single `recv()` returns everything requested even once all the bytes have arrived — a legitimate client's packet can arrive split across reads. When that happened, the old code silently returned a truncated, zero-padded buffer as if it were the whole packet. Whatever garbage length/id fields ended up in that truncated data then got deserialized as real — e.g. a bogus string-length field feeding a `std::string`/`std::vector` construction — and eventually threw (`std::bad_alloc`/`std::length_error`) somewhere downstream, wherever that garbage happened to get used. Fixed by looping `recv()` until the full declared `size` is actually read (or the connection errors/closes), mirroring `sendPacket`'s own existing loop-until-fully-sent pattern just below it in the same file.

**Defense in depth**: nothing anywhere caught exceptions during packet processing, so *any* unexpected exception — not just ones caused by the read bug above — took the entire process down with it, disconnecting every player. `ConnectionManager::processConnection` now wraps its per-connection work in a `try`/`catch`: an exception during normal processing force-closes just that one connection (new `Connection::forceClose()`, directly closes the socket rather than `disconnect()`'s cooperative queue-a-packet-and-wait approach) and lets the *next* call run the same disconnect cleanup a graceful client-initiated close already goes through — including announcing the leave, chunk-viewer cleanup, and saving player data. If the disconnect cleanup itself is what throws (as `PlayerDataPersistence::save` did in one of the three crashes), that path gives up on the connection outright instead of retrying forever against a cleanup that will just keep failing the same way.

Verified against a real, isolated instance (built binary + `server.jar` copied to a scratch directory, self-extracting its own `registry_data`/`server.properties`/world on first run — not the user's actual save): a deliberately truncated Handshake packet (claims a 200-byte server-address string, sends only 10 bytes of it) reliably reproduced the exact `std::bad_alloc` from the log, now caught and logged (`"...closing this connection rather than crashing the server: std::bad_alloc"`) instead of aborting; the server process stayed alive throughout, and a subsequent well-formed Status ping got a real 183-byte response, confirming the server was still fully functional afterward, not just "still running."
## Sand/gravel gravity (block updates)

Doesn't add a general block-update/redstone-style propagation system — nothing else in this project needs one yet, and this feature only needs a narrow, targeted trigger: whenever a block is broken or placed, check whether a gravity-affected block (sand/gravel) is now resting on nothing, and if so pull it out of the static world and simulate it as a real falling entity, mirroring how dropped items already work (`ItemEntityManager`/`ItemPhysicsSystem`).

**`SAND_BLOCK_STATE_ID` (112) / `GRAVEL_BLOCK_STATE_ID` (118)** added to `BlockIds.hpp` — the only two `BlockTable` entries that need a named constant, since the gravity check needs a direct ID comparison rather than a name lookup.

**`CheckGravityBlock(World&, x, y, z)`** (new free function, `network/packets/Play.hpp`/`.cpp`) is the actual trigger: reads the block at `(x,y,z)`; if it's sand/gravel and the block directly below it is air, removes it from the chunk (`World::setBlock` to air, broadcasting `Block_Update_p`), spawns a `FallingBlockEntity`, and broadcasts `Spawn_Entity_p` for it. It then **recurses on `(x, y+1, z)`** — the position it just vacated might itself have been holding up another gravity block directly above, so a multi-block stack collapses one block at a time within the same call, mirroring vanilla's chain-reaction collapse. Called from two places: `Player_Action_p`'s break handler, checking the block above whatever was just broken; `Use_Item_On_p`'s place handler, checking the just-placed block itself (so dropping sand into open air over a hole starts falling immediately, matching vanilla). World floor (`WORLD_MIN_Y`) is treated as solid support, matching `ItemPhysicsSystem`'s own "can't check below the bottom of the world" convention.

**`minecraft:falling_block`'s entity-type ID (40)** confirmed via a freshly-generated `server.jar --reports` (`registries.json`'s `minecraft:entity_type`), cross-checked against the already-known item (58) and player (128) entity-type IDs in the same report. **The Spawn Entity packet's `Object Data` field carries the block state ID directly** for this entity type — confirmed against a version-pinned `minecraft.wiki` revision explicitly labeled "1.21.7" (the Object Data page, not the live page), not assumed from general protocol knowledge. `Spawn_Entity_p` gained an optional `data` parameter (defaults to `0`, preserving every existing call site) to carry this. No entity-metadata packet is sent for falling blocks at all: their only metadata field (index 8, "spawn position") is optional per the protocol's own "not required to send all metadata fields" rule, and this project doesn't use it for anything, so it's simply omitted — the block's appearance comes entirely from `Object Data`.

**`FallingBlockEntity`/`FallingBlockEntityManager`** (new, `entities/`) mirror `ItemEntity`/`ItemEntityManager` exactly, simplified to vertical-only motion (`vy` only — nothing in this project ever pushes a falling block sideways, unlike a tossed item).

**`FallingBlockSystem`** (new `TickSystem`, registered in `TickLoop`) mirrors `ItemPhysicsSystem`'s ground-collision approach (same `GRAVITY = 0.04` blocks/tick² constant, same "check the block one step below the candidate position" collision test) but simpler in two ways: no horizontal drag/velocity (vertical-only), and position corrections (`Teleport_Entity_p`) are sent every tick rather than only past a velocity-change threshold — falling blocks are short-lived (a handful of ticks in the common case), so there's no lengthy horizontal arc for a client to extrapolate and no jitter concern worth throttling for. On landing (block found directly below), the entity converts back into a real block via `World::setBlock` + `Block_Update_p`, and is removed (`Remove_Entities_p`).

Verified offline via a scratch program: breaking the support under a 2-tall sand stack correctly collapses both blocks into two independent falling entities (cascade recursion); placing sand directly onto open air immediately spawns a falling entity; running `FallingBlockSystem::onTick` in a loop against a manually-spawned entity correctly lands it on the expected block and converts it back to a real, static sand block, removing the entity. `Set_Creative_Mode_Slot_p`/`Use_Item_On_p`-triggered placement and the full visual fall need a real client to exercise end-to-end (no offline precedent for this in the project) — verified via real client instead: break the block under a sand/gravel column and watch it fall and land correctly; place sand over a hole and confirm it falls immediately; confirm a landed block is solid/persists across a chunk reload.
